# 説明書（管理者・経営層向け）

## 1) 目的
本書は、本番運用日における勤怠システムの運用手順を定義し、多人数同時利用時のリスクを最小化するための管理者向け手順書です。

## 2) システム情報
- 公式ドメイン: `https://tosouapp.com`
- 生存確認: `https://tosouapp.com/ping`
- 稼働バージョン確認: `https://tosouapp.com/version`

## 3) 役割分担
- 運用管理者:
  - 始業時の打刻状況を監視
  - 障害時の初動判断と連絡統制
- IT/システム担当:
  - ログ、性能、デプロイ状況の確認
  - 回避策および復旧対応の実施

## 4) 高負荷日の運用シナリオ
### 始業30分前
1. `ping` と `version` を開き、稼働状態を確認する。
2. 管理者アカウントでログイン確認を行う。
3. 以下画面を簡易チェックする:
   - `部門管理`
   - `簡易登録画面`
   - `月次勤怠入力`

### 始業時間帯
1. まず 20-30 名を先行利用させる。
2. 最初の 10-15 分を重点監視する:
   - ログイン成功率
   - `開始打刻` の応答速度
   - ログ上の 4xx/5xx 異常
3. 問題がなければ全社員へ展開する。

## 5) Go / No-Go 判定基準
- Go:
  - `ping` が安定している
  - サーバーエラーが増加していない
  - `開始打刻` が正常に完了している
- 一時 No-Go:
  - 広範囲でログイン障害が発生
  - 打刻失敗率が急増
  - 応答遅延が継続的に悪化

## 6) 障害時の一次対応フロー
1. 影響範囲を切り分ける（単一利用者 / 一部部署 / 全体）。
2. 利用者へ一次案内を実施する:
   - `再読込` -> `Ctrl + F5` -> 再ログイン
3. IT担当が以下を確認する:
   - `ping`、`version`、サービスログ
4. 広範囲障害時は:
   - 打刻を時間帯ごとに分散実施
   - 社内へ進捗と復旧見込みを周知

## 7) データ保全ルール
- 承認なしで本番DBへ直接書き込み処理を実行しない。
- 健全性確認は read-only（`GET`）を優先する。
- データ修正は責任者承認と記録を必須とする。

## 8) 社内告知テンプレート（短文）
`勤怠システムは通常稼働中です。各自のアカウントでログインし、始業時に開始打刻、終業時に終了打刻を実施してください。異常が発生した場合は、発生時刻と画面キャプチャを添えて管理者へ連絡してください。`
